## 1 使用说明

### 1.1 列字段定义

| 列名 | 说明 |
|------|------|
| 故障对象 | 发生故障的对象（如主线程、UIAbility等） |
| 故障模式 | 故障对象出现故障的具体方式 |
| 故障原因 | 导致该故障的具体原因或行为 |
| 故障影响 | 该故障导致的更高级别故障或外部表现 |
| 故障检测机制 | 如何检测该故障 |
| 改进措施 | 针对该故障的改进建议 |

### 1.2 故障传播关系

- **故障原因** = 下一级故障模式
- **故障影响** = 上一级故障模式
- **一级故障影响** = 故障场景 + 故障类型

## 3 故障模式详情

### 3.1 主线程

**故障类型**: 业务超时 / 业务阻塞

---

**故障模式1**: 主线程卡死超时

- **故障原因**:
  - 主线程阻塞（等锁、死循环、IO阻塞）
  - 主线程繁忙（大量计算、渲染阻塞）
  - 系统高负载

- **故障影响**: 应用卡死（业务超时）

- **故障检测机制**: 
  - 3s、6s的栈顶都在相同的栈上（去除基础栈查看三层）

- **改进措施**:
  - 避免在主线程进行耗时操作
  - 使用子线程处理耗时任务
  - 合理加锁，避免死锁

---

**故障模式2**: 主线程阻塞

- **故障原因**:
  - 等锁（死锁、锁竞争）
  - 同步IO阻塞
  - 死循环

- **故障影响**: 主线程卡死超时

- **故障检测机制**: 
  - 检测到主线程栈顶长时间不变

- **改进措施**:
  - 使用异步IO
  - 避免死锁（加锁顺序一致）
  - 将耗时计算移到子线程

---

**故障模式3**: 主线程繁忙

- **故障原因**:
  - 大量计算
  - 复杂渲染
  - 大数据处理

- **故障影响**: 主线程卡死超时

- **故障检测机制**: 
  - 主线程CPU使用率持续100%

- **改进措施**:
  - 分批处理大数据
  - 简化渲染逻辑
  - 使用虚拟化列表

---

### 3.2 UIAbility

**故障类型**: 业务超时

---

**故障模式1**: UIAbility超时

- **故障原因**:
  - Load超时（冷启10s）
  - Foreground超时（切前台5s）
  - 应用逻辑错误

- **故障影响**: 应用卡死（业务超时）

- **故障检测机制**: 
  - Load事件超过10s
  - Foreground事件超过5s

- **改进措施**:
  - 优化冷启性能
  - 优化切前台逻辑
  - 减少UIAbility初始化时间
