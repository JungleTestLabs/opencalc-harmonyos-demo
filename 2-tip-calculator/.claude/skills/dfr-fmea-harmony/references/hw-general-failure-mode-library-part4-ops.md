## 1 使用说明

### 1.1 列字段定义

| 列名 | 说明 | 填写要求 |
|------|------|----------|
| 一级对象 | 发生故障的对象类别 | 必填 |
| 二级对象 | 故障发生的具体子系统或组件 | 必填 |
| 三级对象 | 具体的故障位置或场景（可选） | 选填 |
| 四级对象 | 进一步细化的故障位置（可选） | 选填 |
| 故障模式 | 故障对象可能出现故障的方式 | 必填 |
| 严酷度 | I级关键/Ⅱ级重要/Ⅲ级一般/Ⅳ级提示 | 选填 |
| 发生频度 | 1-10，数字越大越容易发生 | 选填 |
| 可检测度 | 1-10，数字越大越难检测 | 选填 |
| SIL等级 | 安全完整性等级 | 选填 |
| 故障原因（对象） | 引发故障的对象 | 选填 |
| 故障原因（行为） | 引发故障的具体行为 | 选填 |
| 故障影响（对象） | 受到故障影响的对象 | 选填 |
| 故障影响（详情） | 故障产生的具体影响描述 | 选填 |
| 改进措施 | 针对该故障模式的改进建议 | 选填 |

### 1.2 RPN评分

| 维度 | 范围 | 说明 |
|------|------|------|
| 严重度S | 1-10 | 后果严重程度，10最严重 |
| 发生率O | 1-10 | 发生可能性，10最可能 |
| 探测度D | 1-10 | 探测难度，10最难 |

### 1.3 RPN阈值

| RPN | 风险 | 处理 |
|-----|------|------|
| ≥100 | 高 | 必须改进措施 |
| 60-99 | 中 | 建议改进措施 |
| <60 | 低 | 可接受监控 |

#### 3.2.3 网络连接

**故障模式 1**: 物理层中断


**故障原因（对象）**:
待定（FMEA分析确定对象）

**故障原因（行为）**:
网络连接中断，或网卡硬件故障

**故障影响（对象）**:
待定（FMEA分析确定对象）

**故障影响（详情）**:
物理连接中断，相关业务中断

**改进措施**:
组网上对处于核心/骨干/汇聚层网络中的关键链路及其他影响用户数量较大的链路提供冗余保护，或采用其他保护机制，确保单个链路中断不会导致整个网络系统瘫痪。
有网络级保护时，及时检测和倒换。上下层配置联动倒换，不能出现反复。无网络级倒换时，及时告警。
链路冗余可以通过设备之间的直连链路冗余，也可以是通过其他网络节点迂回实现，或者通过VRRP、HSRP、FRR、NSR等网络保护协议提供端到端的路径保护。

当检测到性能劣化超过阈值，及时进行线路倒换。
有网络级保护时，及时检测和倒换。上下层配置联动倒换，不能出现反复。无网络级倒换时，及时告警。
链路冗余可以通过设备之间的直连链路冗余，也可以是通过其他网络节点迂回实现，或者通过VRRP、HSRP、FRR、NSR等网络保护协议提供端到端的路径保护。

**故障模式 7**: 脑裂


**故障原因（对象）**:
待定（FMEA分析确定对象）

**故障原因（行为）**:
网络出现故障时（出现网络分区），集群中可能多个节点都认为自己是主节点，都接管服务。

**故障影响（对象）**:
待定（FMEA分析确定对象）

**故障影响（详情）**:
多个节点都认为自己是主节点，都接管服务，同时访问存储，可能导致数据损坏。

**改进措施**:
添加冗余的心跳线，例如双条线。尽量减少“脑裂”发生机会。
对集群系统访问的数据使用“锁”机制，正在服务一方锁住数据，“脑裂”发生时，让对方完全“抢不走”访问权利。
使用“第三方仲裁”或“多数派仲裁”机制，一旦出现脑裂，能够通过仲裁机制确定主节点。
