# KEP（关键体验路径）验证规则

本文档定义 prd-generator SKILL 中关键体验路径 (KEP) 的验证规则。

## KEP 表格格式

### 终端V2.1 PRD 模板中的 KEP 表格

KEP信息以表格形式呈现，位于 `1.7.5 关键体验路径（KEP）` 章节：

```markdown
| 阶段 | 场景类型 | 涉及Actor | KEP描述 | 场景名称 | 场景编号 | 用例名称 | 用例编号 | IR需求编号 |
|---|---|---|---|---|---|---|---|---|
| XXX | XX | XXX | | | | | | |
```

### KEP 表格字段说明

| 字段 | 说明 | 必需 | 示例 |
|------|------|------|------|
| 阶段 | 用户旅程的阶段 | 否 | 发现、体验、使用、分享 |
| 场景类型 | 场景分类 | 否 | 正常流程、异常流程、边界情况 |
| 涉及Actor | 参与者/用户角色 | 否 | 最终用户、管理员 |
| KEP描述 | KEP的详细描述 | **是** | 用户查看磁盘信息 |
| 场景名称 | 场景的简短名称 | 推荐 | 查看磁盘列表 |
| 场景编号 | 场景的唯一标识 | 推荐 | SC001 |
| 用例名称 | 对应的测试用例名称 | 推荐 | 查看磁盘列表用例 |
| 用例编号 | 用例的唯一标识 | 推荐 | TC001 |
| IR需求编号 | 关联的需求编号 | 推荐 | IR-001 |

## KEP 命名规范

### KEP 描述规则

1. **动词优先**：使用中文动词开头
2. **简洁清晰**：描述核心行为
3. **用户视角**：从用户角度描述

### 示例

| 推荐 | 不推荐 | 原因 |
|------|--------|------|
| 查看磁盘信息 | 磁盘信息查看 | 不是动词优先 |
| 格式化磁盘 | 磁盘格式化操作 | 多余的词语 |
| 设置磁盘名称 | 修改 | 太模糊 |
| 执行磁盘修复 | 修复磁盘 | 良好 |

### 常用动词

| 类别 | 动词 |
|------|------|
| 查询类 | 查看、获取、显示、列举、浏览 |
| 操作类 | 执行、启动、停止、暂停、继续 |
| 修改类 | 设置、修改、更新、变更、编辑 |
| 删除类 | 删除、移除、清除、卸载 |
| 创建类 | 创建、新增、添加、插入 |

## KEP 完整性验证

### 必需字段检查

每个 KEP 记录必须包含：

- [ ] **KEP描述**：清晰描述用户行为
- [ ] **场景编号**：唯一标识符
- [ ] **用例编号**：关联测试用例
- [ ] **IR需求编号**：关联需求

### KEP 数量要求

| 项目类型 | 最少KEP | 推荐P0 KEP |
|----------|---------|-----------|
| 主要功能 | 3-5 | 2-3 |
| 次要功能 | 2-3 | 1-2 |
| 增强功能 | 1-2 | 1 |

### 场景覆盖检查

- [ ] **正常流程**：覆盖主要用户路径
- [ ] **异常流程**：覆盖错误处理路径
- [ ] **边界情况**：覆盖极端情况

## KEP 与 KEI 的关联

### KEI（关键体验指标）格式

KEI信息以表格形式呈现，位于 `1.7.6 关键体验指标（KEI）` 章节：

```markdown
| 评估体系 | 维度 | 评估类型 | 是否必选 | 评估标准 | IR需求编号 | 功能名称 | 功能编码 |
|---|---|---|---|---|---|---|---|
```

### KEI 字段说明

| 字段 | 说明 | 示例 |
|------|------|------|
| 评估体系 | 评估框架名称 | 用户体验、性能、稳定性 |
| 维度 | 评估的具体维度 | 响应时间、操作便捷性 |
| 评估类型 | 评估方法类型 | 定量、定性 |
| 是否必选 | 是否为必选指标 | 是/否 |
| 评估标准 | 具体的评估数值或描述 | <2秒、成功率>99% |
| IR需求编号 | 关联的需求编号 | IR-001 |
| 功能名称 | 对应的功能名称 | 查看磁盘信息 |
| 功能编码 | 功能的唯一编码 | F001 |

### KEP-KEI 关联规则

每个 KEP 应该有对应的 KEI 指标：

1. **性能指标**：响应时间、加载速度等
2. **体验指标**：操作步骤、交互便捷性等
3. **质量指标**：成功率、错误率等

## KEP 验证检查清单

对 PRD 中的每个 KEP，验证：

- [ ] KEP描述以动词开头
- [ ] KEP描述简洁明确（≤20字）
- [ ] 场景编号唯一
- [ ] 用例编号唯一
- [ ] IR需求编号存在
- [ ] 至少覆盖正常流程
- [ ] 覆盖主要异常流程
- [ ] 有对应的KEI指标

## 常见验证错误

### 错误 1：KEP描述模糊

**问题**：KEP描述不够具体

**示例**：
- ❌ "磁盘操作"
- ✅ "查看磁盘信息"

**解决方案**：使用更具体的动词和对象

### 错误 2：缺少场景编号

**问题**：KEP记录没有场景编号

**影响**：无法追踪和引用

**解决方案**：为每个KEP分配唯一场景编号

### 错误 3：KEP与需求未关联

**问题**：KEP没有对应的IR需求编号

**影响**：无法追溯需求来源

**解决方案**：确保每个KEP关联到至少一个需求

### 错误 4：缺少KEI指标

**问题**：KEP没有对应的体验指标

**影响**：无法量化评估体验

**解决方案**：为关键KEP添加KEI指标

## KEP 覆盖分析

验证单个 KEP 后，分析整体覆盖：

1. **阶段覆盖**：是否覆盖用户旅程的各个阶段？
2. **角色覆盖**：是否覆盖所有用户角色？
3. **场景覆盖**：是否覆盖正常、异常、边界场景？

### 覆盖指标

| 指标 | 目标 | 说明 |
|------|------|------|
| KEP 数量 | ≥3 | 最少关键用户路径 |
| 用户角色覆盖 | 所有定义的角色 | 每个角色应有相关 KEP |
| 正常路径覆盖 | 100% | 覆盖所有正常操作 |
| 异常路径覆盖 | ≥80% | 覆盖大多数错误场景 |
| KEI 关联率 | ≥90% | 至少90%的KEP有KEI |

## KEP 生成建议

### 从需求生成 KEP

1. **识别用户目标**：用户想要达成什么？
2. **分解用户旅程**：分阶段描述
3. **定义关键路径**：找出最重要的路径
4. **添加场景编号**：为每个场景分配编号
5. **关联测试用例**：确保可测试性
6. **定义体验指标**：量化评估标准

### KEP 描述模板

```
[动词] + [对象] + [条件/修饰]
```

**示例**：
- 查看磁盘信息
- 格式化磁盘（快速模式）
- 修复磁盘（自动检测）

## 自动验证脚本示例

```python
import re

def validate_kep_table(table_content):
    """验证KEP表格的完整性"""
    issues = []

    # 检查必需列
    required_columns = ['KEP描述', '场景编号', '用例编号', 'IR需求编号']
    for col in required_columns:
        if col not in table_content:
            issues.append(f"缺少必需列: {col}")

    # 检查KEP描述格式
    kep_pattern = r'^(查看|获取|显示|列举|执行|启动|停止|设置|修改|更新|删除|移除|创建|新增|添加)'
    for row in table_content:
        kep_desc = row.get('KEP描述', '')
        if kep_desc and not re.match(kep_pattern, kep_desc):
            issues.append(f"KEP描述格式不规范: {kep_desc}")

    return issues
```
