# rq-clarify — 需求澄清

> 通过交互式问答，将模糊需求转化为清晰、完整、可执行的需求描述。

## 定位

好的软件需求应该是"无歧义的"。但现实中，用户很少一开始就能把需求说清楚。rq-clarify 的职责就是通过恰当的提问，把这些"默认值"显性化。

核心原则：**问得少，问得准**。不是把所有可能的问题都抛出来，而是只问那些真正影响质量的关键问题。

## 输入

- 用户提供的需求描述（可以是模糊的）
- 或 `rq-parse` 的结构化分析结果（包含模糊点列表）

## 澄清策略

### 提问优先级

```
blocker 模糊点（必须澄清）
  ↓ 优先处理
important 模糊点（影响质量，尽量澄清）
  ↓ 次优先
nice-to-have 模糊点（告知用户，不强制追问）
```

### 提问批次控制

- **每轮最多 3-5 个问题**
- **相关问题合并**：多个模糊点指向同一个功能区域时，合并为一个综合问题
- **依赖问题串联**：问题 B 的答案取决于问题 A 时，先问 A
- **提供选项而非开放题**：尽量给出可选方案让用户选择

### 提问模板

#### 范围模糊 → 边界界定法

```
你提到"XX功能"，我理解可能包括以下几个方面：
1. [方面A] — 例如...
2. [方面B] — 例如...
3. [方面C] — 例如...

本次需求是要覆盖哪些？有没有明确不做的？
```

#### 行为模糊 → 场景还原法

```
关于[某个行为]，我想确认几个场景下的预期表现：
- 场景1：当用户[做某事]时，应该发生什么？
- 场景2：如果[某个条件]不满足，该怎么处理？
- 场景3：[边界情况]下的预期行为是？
```

#### 交互模糊 → 原型描述法

```
关于[某个交互]，目前有两种常见的实现方式：
- 方案A：[描述交互方式A]，适合[场景]
- 方案B：[描述交互方式B]，适合[场景]

你倾向哪种？或者有其他想法？
```

#### 数据模糊 → 字段枚举法

```
关于[数据实体]需要保存哪些信息，我列了一个初步清单：
- [字段1]：[说明]（必须/可选）
- [字段2]：[说明]（必须/可选）
- ...

有需要增减的吗？
```

#### 约束模糊 → 量化锚定法

```
你提到[某个非功能要求]，我需要一个可衡量的标准：
- 目前的表现大约是[基线]（如果已知）
- 你期望达到的目标是？（例如：响应时间<Xms，文件大小<YMB）
- 有没有硬性的底线要求？
```

#### 兼容模糊 → 影响枚举法

```
这个变更可能影响以下现有功能：
- [功能A]：[可能的影响]
- [功能B]：[可能的影响]

需要确保这些功能的行为不变吗？还是允许部分调整？
```

## HarmonyOS ArkTS 专项澄清

### 数据存储方案选择
```
关于数据存储，这个功能适合用：
- Preferences（键值对）：适合简单配置项、开关、少量设置
- RelationalStore（关系型数据库）：适合结构化数据、需要复杂查询
- 文件存储（fileIo）：适合大文件、二进制数据

你的数据特征更接近哪种？大约有多少条数据？
```

### 离线/在线策略
```
这个功能在没有网络时：
- 完全不可用？（纯在线功能）
- 使用缓存数据，但不能更新？
- 需要支持离线操作，联网后同步？
```

### 后台行为
```
这个功能是否需要在应用退到后台后继续运行？
- 如果需要：是短时任务（几秒到几分钟）还是长时任务（如后台播放）？
- HarmonyOS 的后台任务有严格限制，这会影响实现方案。
```

### 权限需求
```
这个功能可能需要以下系统权限：
- [权限1]：用于[目的]
- [权限2]：用于[目的]

需要确认：用户拒绝权限时的降级方案是什么？
```

## 交互流程

```
接收需求或模糊点列表
    ↓
按优先级排序 → 分批提问
    ↓
第 1 轮提问（blocker 级别，3-5 个问题）
    ↓
← 用户回答
    ↓
分析回答 → 更新需求理解 → 检查是否引入新模糊点
    ↓
[如有未解决的 blocker/important] → 下一轮提问
    ↓
... 重复直到关键模糊点解决 ...
    ↓
生成澄清后需求摘要 → 用户确认
```

### 用户回答处理

- **回答偏离问题**：温和地重新引导
- **回答"我不确定"**：提供合理的默认建议
- **回答引入新需求**：记录下来，但提醒聚焦原始需求
- **用户表示"别问了，你决定"**：基于行业最佳实践做出合理假设

### 提前终止

- 用户明确表示"够了"或"就这样"
- 所有 blocker 模糊点已解决
- 用户要求直接进入下一步

提前终止时，将未澄清的点标记为 `assumption`（假设）。

## 输出格式

```markdown
## 澄清后需求摘要

### 需求概述
[用 2-3 句话清晰完整地描述需求]

### 已澄清的关键决策
| # | 问题 | 决策 | 来源 |
|---|------|------|------|
| 1 | [原始模糊点] | [用户的决定] | 用户确认 |
| 2 | [原始模糊点] | [采纳的方案] | 用户选择方案B |
| 3 | [原始模糊点] | [默认值] | 用户同意默认方案 |

### 待定假设
以下几点未明确，采用了合理默认值（可在实现阶段调整）：
- [假设1]：[默认方案及理由]
- [假设2]：[默认方案及理由]

### 功能边界
- **包含**：[明确列出本次需求包含的功能]
- **不包含**：[明确列出不做的功能]
```

## 示例

### 示例：从模糊到清晰

**原始需求中的模糊点**：
1. [blocker] "离线下载"——下载整个播客系列还是单集？
2. [blocker] 下载的文件存在哪里？用户能管理已下载文件吗？
3. [important] 下载时的网络策略——仅 WiFi 还是允许移动数据？
4. [nice-to-have] 下载文件格式是否需要转换？

**第 1 轮提问**：

> 关于离线下载功能，我需要确认几个关键点：
>
> 1. **下载范围**：用户可以下载单集还是整个播客系列？
>    - A. 仅支持单集下载
>    - B. 支持单集和批量下载（整个系列或多选）
>
> 2. **文件管理**：用户是否需要一个"下载管理"页面来查看、删除已下载的文件？
>
> 3. **网络策略**：下载时的默认网络策略是？
>    - A. 仅 WiFi（推荐，播客文件通常较大）
>    - B. WiFi + 移动数据
>    - C. 让用户自己在设置里选

**用户回答**：
> "1选B，2要的，3选C但默认仅WiFi"

**澄清后摘要**：
> 离线下载功能支持单集下载和批量下载（按系列或多选）。提供下载管理页面用于查看进度、删除已下载文件。网络策略默认仅 WiFi，用户可在设置中切换为 WiFi+移动数据。

## 核心约束

1. **不要替用户做决策**：提供方案但让用户决定
2. **不要过度追问**：nice-to-have 级别的模糊点不强制追问
3. **不要引入技术黑话**：用用户能理解的语言
4. **保持上下文连贯**：每轮提问前简要概括上一轮的结论
