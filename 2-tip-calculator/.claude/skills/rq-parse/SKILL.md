# rq-parse — 需求解析

> 将用户的自然语言需求转换为结构化分析，识别关键实体和技术影响。

## 定位

用户的原始需求往往是非结构化的自然语言。rq-parse 的职责是从中提取出足够的结构化信息：判断用户想做什么、涉及哪些技术领域、影响哪些代码层次、哪些地方说得不清楚需要追问。

## 输入

用户的原始需求文本。可以是：
- 一句话描述："我想给播客添加离线下载功能"
- 场景描述："用户在地铁里没网，想听之前订阅的播客"
- Bug 报告："播放列表切歌时偶尔会卡死"
- 改进建议："能不能把播放速度的选项做得更细一些"
- 模糊想法："播客应用需要更好的发现体验"

## 工作流程

### 意图分类

判断需求属于哪种类型：

| 类型 | 标识 | 特征 | 侧重 |
|------|------|------|-----------|
| 新增功能 | `新增功能` | 当前不存在的能力 | 功能需求 + 交互流程 |
| 功能变更 | `功能变更` | 已有功能的改进 | 变更点 + 兼容性 |
| 缺陷修复 | `缺陷修复` | 现有行为不符预期 | 复现步骤 + 根因 |
| 优化或重构 | `优化或重构` | 架构优化、性能优化、DFX | 影响范围 + 回归风险 |
| 平台迁移 | `平台迁移` | 跨平台迁移 | 兼容性 + 接口适配 |

### 领域映射

将需求映射到 HarmonyOS ArkTS 的技术领域：

| 领域 | 关键词信号 | 典型涉及 |
|------|-----------|---------|
| UI/交互 | 页面、界面、按钮、列表、动画、布局 | `arkts-component-builder`, `arkts-animation-builder` |
| 数据管理 | 存储、数据库、缓存、持久化 | `arkts-data-layer` |
| 网络通信 | 请求、接口、API、下载、上传 | `arkts-data-layer`, `arkts-download-manager` |
| 媒体播放 | 播放、暂停、音频、视频、后台 | `arkts-media-playback` |
| 导航路由 | 页面跳转、Tab、返回、路由 | `arkts-navigation-builder` |
| 状态管理 | 同步、刷新、共享数据、全局状态 | `arkts-state-manager` |
| 系统能力 | 权限、文件、相册、通知、后台任务 | `arkts-system-capabilities` |
| 国际化 | 多语言、翻译、本地化 | `arkts-i18n` |
| WebView | 网页、浏览器、H5、注入 | `arkts-webview-manager` |

### 影响层次分析

基于 MVVM 分层架构，判断需求涉及哪些层次：

```
┌─────────────────────────────────────────┐
│  View 层（pages/, components/）         │  ← UI 结构、交互、样式
├─────────────────────────────────────────┤
│  ViewModel 层（viewmodels/）            │  ← 业务逻辑、状态转换
├─────────────────────────────────────────┤
│  Model 层（models/）                    │  ← 数据模型定义
├─────────────────────────────────────────┤
│  数据层（database/, network/, parser/） │  ← 持久化、网络、解析
├─────────────────────────────────────────┤
│  基础设施（common/, helpers/, workers/）│  ← 工具、常量、后台任务
└─────────────────────────────────────────┘
```

变更类型：
- `new` — 需要新建文件/模块
- `modify` — 修改现有代码
- `extend` — 扩展已有接口/类型

### 关键实体提取

识别具体的技术实体：

- **数据实体**：需要新建或修改的数据模型（如 `DownloadTask`, `PlaybackQueue`）
- **页面实体**：涉及的页面或组件（如 "下载管理页", "播放器底栏"）
- **接口实体**：需要调用或新增的 API（如 RSS 解析、文件下载）
- **事件实体**：需要处理的事件或状态变化（如 "下载完成通知", "播放进度更新"）

### 模糊点检测

标记需求中信息不足或存在歧义的地方：

| 类别 | 示例 | 影响 |
|------|------|------|
| 范围模糊 | "更好的发现体验"——具体指什么？ | 无法确定功能边界 |
| 行为模糊 | "支持离线"——下载策略是什么？ | 无法确定实现方案 |
| 交互模糊 | "添加搜索"——搜索什么？在哪搜索？ | 无法设计 UI |
| 数据模糊 | "保存播放记录"——保存哪些字段？保留多久？ | 无法设计数据模型 |
| 约束模糊 | "要快"——多快算快？有基线吗？ | 无法定义验收标准 |
| 兼容模糊 | "不影响现有功能"——具体指哪些？ | 无法确定回归范围 |

严重程度：
- `blocker` — 不澄清无法进行下一步
- `important` — 影响质量，应尽量澄清
- `nice-to-have` — 可以在实现阶段再确定

### 复杂度评估

| 等级 | 特征 | 典型工作量 |
|------|------|-----------|
| S（简单） | 单一领域、1-2 个文件 | 单个 spec 即可 |
| M（中等） | 2-3 个领域、跨 2-3 层 | 需要拆分子功能 |
| L（复杂） | 多领域联动、需要新建模块 | 需要拆分多个 spec |
| XL（特大） | 架构级变更、影响全局 | 需要分阶段实施 |

## 输出格式

```markdown
## 需求解析报告

### 原始需求
> [用户的原始描述]

### 意图分类
- **类型**：新增功能 | 功能变更 | 缺陷修复 | 优化或重构 | 平台迁移
- **一句话概括**：[用你自己的话概括需求的核心]

### 领域映射
- [领域1]：[简要说明涉及什么]
- [领域2]：[简要说明涉及什么]

### 影响层次
| 层次 | 变更类型 | 说明 |
|------|---------|------|
| View | new/modify/extend | ... |
| ViewModel | ... | ... |

### 关键实体
- 数据实体：[列表]
- 页面实体：[列表]
- 接口实体：[列表]
- 事件实体：[列表]

### 模糊点
| # | 类别 | 描述 | 严重程度 |
|---|------|------|---------|
| 1 | 范围模糊 | ... | blocker |
| 2 | 行为模糊 | ... | important |

### 复杂度评估
- **等级**：S / M / L / XL
- **理由**：[简要说明]

### 备注
- [如果存在 blocker 模糊点] → 建议在继续前澄清这些点

```

## 边界判断

**什么时候需求"足够清晰"可以继续？**
- 没有 `blocker` 级别的模糊点
- 意图分类明确
- 影响范围可以确定

**什么时候应该建议拆分需求？**
- 复杂度达到 L 或 XL
- 涉及 4 个以上技术领域
- 需求中包含多个独立的功能点

## 与项目现有代码的关联

解析需求时，结合项目当前状态提供更精准的分析：

1. **检查现有实现**：读取 `entry/src/main/ets/` 目录结构，判断需求涉及的模块是已存在还是需要新建
2. **检查 spec 目录**：读取 `spec/` 下的现有 spec，判断是否与已有 spec 冲突或重复
3. **参考迁移规格**：如果项目有 `docs/migration-spec.md`，检查需求是否属于已规划但未实现的部分

## 示例

### 示例 1：清晰的功能需求

**用户输入**："给播客添加睡眠定时器，可以设置 15/30/45/60 分钟后自动停止播放"

**解析结果**：
- 意图：`新增功能`
- 领域：媒体播放、UI/交互
- 层次：View（定时器设置 UI）、ViewModel（定时逻辑）、Model（定时配置）
- 关键实体：SleepTimer 模型、定时器设置弹窗、PlaybackService 扩展
- 模糊点：无 blocker
- 复杂度：S

### 示例 2：模糊的需求

**用户输入**："播客应用需要更好的发现功能"

**解析结果**：
- 意图：`功能变更`（但可能是 `新增功能`，需澄清）
- 领域：UI/交互、网络通信、数据管理
- 模糊点：
  1. [blocker] "发现功能"具体指什么？推荐？搜索？分类浏览？热门排行？
  2. [blocker] 数据源从哪来？有后端 API 还是需要聚合 RSS 目录？
  3. [important] "更好"的标准是什么？现有发现功能的问题在哪？
- 复杂度：M-L（取决于澄清结果）
