# PRD 文档格式规范（终端V2.1）

本文档规定基于终端V2.1模板的PRD文档格式要求。

## 文档头部

每个 PRD 必须以以下内容开头：

```markdown
# 产品需求文档（PRD）模板_终端V2.1

## 文档信息

| 项目 | 内容 |
|------|------|
| 文档名称 | 产品需求文档（PRD）模板 |
| 版本号 | V2.1 |
| 创建日期 | [日期] |
| 最后更新 | [日期] |
| 文档状态 | [草稿/评审中/已批准] |
```

## 必需章节

### 1. 文档变更记录

| 版本 | 日期 | 修改人 | 修改内容 | 备注 |
|------|------|--------|----------|------|
| V2.1 | [日期] | [姓名] | [修改描述] | [备注] |

### 2. 产品需求

#### 2.1 需求来源
- 描述需求提出部门及接口人
- 需求分类（运营需求、优化需求、用户反馈需求、规划需求）

#### 2.2 需求背景及业务定位
- 描述需求的原始诉求
- 本需求需要解决的问题或期望达到的诉求

#### 2.3 需求价值分析
- 描述需求给产品带来的价值
- KPI提升、用户体验改善、用户满意度提升、减少运营工作量等

#### 2.4 需求适用范围分析
- 产品品类覆盖（手机直板机、折叠机、平板、PC、智慧屏、座舱、手表等）
- 发布范围（中国区、亚非拉、欧洲等）

#### 2.5 目标用户分析（可选）
- 目标用户画像
- 用户基本信息
- 产品/服务使用情况及使用经验
- 动机与关注点
- 需求及痛点

#### 2.6 竞品分析（可选）
- 竞品功能对比表格
- 优劣势总结
- 竞争力构建计划

#### 2.7 需求描述

##### 1.7.1 需求参与者（角色）
- 最终用户

##### 1.7.2 前置条件
- 设备要求
- 系统版本要求
- 验收范围

##### 1.7.3 业务流程
- 正常业务流程
- 异常业务流程

##### 1.7.4 业务功能描述
- 功能表格（标题、IR需求编号、描述、功能名称、功能编码、备注）

##### 1.7.5 关键体验路径（KEP）
- KEP表格（阶段、场景类型、涉及Actor、KEP描述、场景名称、场景编号、用例名称、用例编号、IR需求编号）

##### 1.7.6 关键体验指标（KEI）
- KEI表格（评估体系、维度、评估类型、是否必选、评估标准、IR需求编号、功能名称、功能编码）

##### 1.7.7 公共需求/能力交互影响分析

**整体分析（必选）**：
- 国际化（中国大陆、全球）
- 安全隐私（儿童账号、已收集个人信息清单上报、隐私空间、停止服务）
- 全局业务（无障碍、适老化、退账号、注销账号、OOBE、克隆备份、旋转屏、分屏、外屏适配、自由多窗、双升单、三方应用关联启动管理、单回退双、小艺问答操控、统一交互）

**受影响的相关功能分析（可选）**：
- 具体描述哪些功能需要支持上述能力/要求

##### 1.7.8 运营策略
- 产品/特性/需求名称
- 用户触点（整体、端内、门店体验及演示、其他触点）
- 运营目标
- 运营策略

### 4. 非功能需求

#### 4.1 性能要求
- 响应时间
- 并发用户数
- 数据处理量

#### 4.2 安全要求
- 数据加密
- 权限控制

#### 4.3 兼容性要求
- 平台版本要求

### 12. 附录

#### 12.1 术语表
- 术语定义

#### 12.2 参考资料
- 相关文档列表

#### 12.3 相关文档
- 文档名称列表

|------|------|------|------|

## KEP 格式规范

### KEP ID 格式

KEP ID 应遵循以下格式：
- **场景编号**：数字编号
- **用例编号**：数字编号

### KEP 表格字段

| 字段 | 说明 | 示例 |
|------|------|------|
| 阶段 | 需求阶段 | 需求分析 |
| 场景类型 | 场景分类 | 正常流程 |
| 涉及Actor | 参与者 | 最终用户 |
| KEP描述 | 关键体验路径描述 | 用户完成XX操作 |
| 场景名称 | 场景名称 | XX功能使用 |
| 场景编号 | 场景唯一编号 | SC001 |
| 用例名称 | 测试用例名称 | XX功能测试 |
| 用例编号 | 用例唯一编号 | TC001 |
| IR需求编号 | 关联的需求编号 | IR001 |

## KEI 格式规范

### KEI 表格字段

| 字段 | 说明 | 示例 |
|------|------|------|
| 评估体系 | 评估框架 | 用户体验 |
| 维度 | 评估维度 | 易用性 |
| 评估类型 | 评估分类 | 定量/定性 |
| 是否必选 | 是否必须评估 | 是 |
| 评估标准 | 具体标准 | 满意度≥4.5 |
| IR需求编号 | 关联的需求编号 | IR001 |
| 功能名称 | 功能名称 | XX功能 |
| 功能编码 | 功能编码 | F001 |

## 格式指南

### 语言
- 使用简体中文
- 技术术语可在括号中包含英文

### 编号
- 主要章节使用中文编号（一、二、三、）
- 子章节使用十进制编号（1.1、1.2）

### 表格
- 使用 Markdown 表格格式
- 包含表格标题以保持清晰

### 代码块
- 使用带语言标识符的三个反引号
- 指定语言（python、json、sql 等）

## 版本控制

### 版本编号
- 格式：V{主版本}.{次版本}
- 主版本：重大功能增加
- 次版本：错误修复、澄清说明

### 变更记录
在文档开头维护变更记录，包括版本、日期、修改人、修改内容、备注。

## 验证检查清单

提交 PRD 前，验证：

- [ ] 文档信息完整（文档名称、版本号、创建日期、最后更新、文档状态）
- [ ] 文档变更记录存在
- [ ] 需求来源已描述
- [ ] 需求背景及业务定位清晰
- [ ] 需求价值分析明确
- [ ] 需求适用范围分析完整（产品品类、发布范围）
- [ ] 需求参与者（角色）已定义
- [ ] 前置条件已明确
- [ ] 业务流程已描述（正常、异常）
- [ ] 业务功能描述完整
- [ ] KEP表格已填写
- [ ] KEI表格已填写
- [ ] 公共需求/能力交互影响分析（整体分析）已完成
- [ ] 运营策略已定义
- [ ] 非功能需求已描述（性能、安全、兼容性）
- [ ] 附录完整（术语表、参考资料、相关文档）
- [ ] 审批记录已预留
