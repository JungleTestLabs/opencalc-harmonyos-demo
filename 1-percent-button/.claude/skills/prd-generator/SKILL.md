# PRD Generator

基于终端V2.1 PRD模板的PRD文档生成与分析工具，支持完整的PRD生命周期管理。

## 快速开始

### 生成PRD文档

```
生成PRD: {需求描述}
- 产品名称: xxx
- 文档版本: V1.0
- 创建日期: 2026-02-10
```

或使用交互式生成：

```
创建一个新的PRD文档
```

### 分析PRD文档

```
分析PRD: {PRD文件路径}
```

或指定具体选项：

```
分析PRD: {PRD文件路径}
- 输出格式: JSON
- 检查完整性: 是
- 提取KEP: 是
- 检查KEI: 是
- 验证公共需求: 是
```

## PRD生成功能

### 1. 基础信息生成

自动生成以下章节：
- **文档信息**：文档名称、版本号、创建日期、文档状态
- **文档变更记录**：版本历史追踪，支持自动更新变更记录

#### 1.1 文档变更记录自动更新

**功能说明**：
当修改 PRD 文档时，自动更新文档变更记录，包括：
- 版本号自动递增（V1.0 → V1.1 → V1.2...）
- 最后更新日期自动更新为当前日期
- 自动添加变更记录条目

**触发条件**：
- 使用 SKILL 修改 PRD 文档内容时
- 检测到文档内容变化时

**变更记录内容**：
```
| 版本 | 日期 | 修改人 | 修改内容 | 备注 |
|------|------|--------|----------|------|
| V{新版本号} | {当前日期} | 产品团队 | {修改内容摘要} | - |
```

**修改内容描述规范**：
- 使用清晰的动词开头（如"优化"、"修正"、"新增"、"删除"等）
- 多个修改点用分号或编号分隔
- 重要的修改项应突出说明

**示例**：
```
| V1.5 | 2026-02-11 | 产品团队 | 优化界面交互：1) 新增日程快速创建功能；2) 优化导航按钮位置；3) 添加日程提醒设置 | - |
```

**使用方式**：
```
更新PRD: {PRD文件路径}
- 修改内容: {修改摘要}
```

或直接修改文档后自动记录变更：
```
分析并更新PRD: {PRD文件路径}
```

## PRD分析功能

### 1. 文档元数据解析

提取以下信息：
- 文档版本（VX.X 格式）
- 创建日期（YYYY-MM-DD）
- 文档作者/团队
- 文档状态
- 产品名称

### 2. KEP提取与验证

对每个找到的KEP，提取并验证：
- **KEP描述**：具体场景描述
- **场景类型**：正常/异常/边界
- **涉及Actor**：参与者
- **场景编号**：唯一标识
- **用例名称**：测试用例名称
- **用例编号**：测试用例编号
- **IR需求编号**：关联需求编号

### 3. KEI提取与验证

提取关键体验指标：
- **评估体系**：性能/体验/功能等
- **维度**：具体评估维度
- **评估类型**：定量/定性
- **是否必选**：是/否
- **评估标准**：具体标准值
- **IR需求编号**：关联需求编号
- **功能名称**：对应功能
- **功能编码**：功能编码

### 4. 完整性验证

使用 [references/completeness_rules.md](references/completeness_rules.md) 中的检查清单验证所有必需章节。

7. 公共需求/能力交互影响分析
8. 运营策略
9. 非功能需求

11. 审批记录

### 9. 规范性检查

#### 9.1 收集个人信息清单检查

使用 [references/personal_info_collection.md](references/personal_info_collection.md) 中的规则验证：

**检查内容**：
- 是否需要提供《收集个人信息清单》
- 《收集个人信息清单》内容是否完整
- 敏感个人信息是否突出明示
- 是否方便用户查阅（一级或二级菜单入口）
- 隐私声明中是否包含访问链接

**落地范围判断**：
- 有桌面图标的应用：
  - 个人数据上云 → 必须提供
  - 不上云但涉及工信部关注/敏感个人信息 → 必须提供
  - 不上云也不涉及 → 不需要
- 无桌面图标的应用：
  - 不上架应用市场 → 不需要
  - 上架应用市场 → 同上

**已收集个人信息清单检查**（强制产品）：
- 终端云：浏览器、应用市场、华为视频、华为音乐、华为阅读、游戏中心、教育中心、华为主题、智慧助手今天、华为钱包
- 软件部：运动健康、畅连、智慧生活
- 电商：华为商城
- MSS：我的华为

#### 9.2 未成年人保护检查

使用 [references/minor_protection.md](references/minor_protection.md) 中的规则验证：

**检查内容**：
- 是否涉及未成年人用户（0-17岁）
- 是否涉及儿童用户（0-14岁）
- 是否提供个性化广告/直接营销
- 基本功能模式是否支持
- 未成年人模式是否适配
- 华为账号年龄段获取是否正确

**年龄分段要求**：
- 儿童（0-14岁）：全面禁止个性化广告/直接营销，个性化推荐默认关闭
- 未成年人（15-17岁）：禁止个性化广告/直接营销，个性化推荐保持现状
- 不得通过自动化决策方式向未成年人进行商业营销

#### 9.3 国际化界面用户规范检查

使用 [references/internationalization_ui.md](references/internationalization_ui.md) 中的规则验证：

**检查内容**：
- 中国大陆发布是否支持6种语言（中文/英文/繁体中文/藏文/维文）
- 全球发布是否满足全球化规范
- 多语言切换功能是否完整
- 文本资源是否正确提取
- UI布局是否适配不同语言

**输出**：生成规范性检查报告，包含：
- 检查项列表
- 检查结果（通过/不通过）
- 问题说明
- 修复建议

### 10. 自动变更记录管理

当更新 PRD 文档时，自动生成和维护文档变更记录。

#### 10.1 变更记录自动生成规则

**触发条件**：
- 修改 PRD 文档内容（章节、功能、需求等）
- 修正文档格式或规范性问题
- 添加或删除功能描述
- 更新公共需求分析

**自动更新内容**：
1. 更新文档信息
   - 版本号递增（V1.0 → V1.1 → V1.2...）
   - 更新最后更新日期为当前日期

2. 添加变更记录
   - 在变更记录表格顶部添加新版本记录
   - 格式：`| V{新版本号} | {日期} | {修改人} | {修改内容} | - |`
   - 修改内容应简洁明了，说明主要修改点

#### 10.2 变更记录格式规范

**变更记录表格格式**：
```markdown
| 版本 | 日期 | 修改人 | 修改内容 | 备注 |
|------|------|--------|----------|------|
| V1.4 | 2026-02-10 | 产品团队 | 修正日程同步方式：将云同步改为近端同步，更新异常业务流程、克隆备份说明、数据准确性要求 | - |
| V1.3 | 2026-02-10 | 产品团队 | 修正座舱特性支持：1) 座舱不支持无障碍功能；2) 语言支持仅中英文2种语言 | - |
| V1.2 | 2026-02-10 | 产品团队 | 按照终端V2.1模板规范修正：1) 章节编号修正（需求描述改为第3章）；2) KEP描述改为动词优先格式 | - |
```

**修改内容描述规范**：
- 使用清晰的动词开头（修正、更新、添加、删除、优化等）
- 多个修改点用分号或编号分隔
- 重要的修改项应突出说明
- 描述应简洁明了，不超过100字

**示例**：
```
修正日程同步方式：将云同步改为近端同步，更新异常业务流程、克隆备份说明、数据准确性要求
```

#### 10.3 版本号管理规则

**版本号格式**：V{主版本}.{次版本}

**递增规则**：
- **次版本递增**（V1.0 → V1.1 → V1.2）：
  - 小幅修改（错别字、格式调整）
  - 补充缺失内容
  - 修正描述错误
  - 添加少量功能

- **主版本递增**（V1.0 → V2.0）：
  - 重大需求变更
  - 架构调整
  - 大量功能增删
  - 产品方向改变

#### 10.4 使用方式

**更新 PRD 时自动生成变更记录**：
```
更新 {PRD文件路径} 的 PRD 文档
- 修改内容: {修改描述}
```

**示例**：
```
更新 日历车机需求PRD.md 的 PRD 文档
- 修改内容: 修正日程同步方式，将云同步改为近端同步
```

**输出**：
- 自动递增版本号
- 自动更新最后更新日期
- 自动添加变更记录到变更记录表格顶部
- 提示用户变更记录已生成

#### 10.5 变更记录检查

在分析 PRD 时，检查变更记录的完整性：

**检查内容**：
- 变更记录表格是否存在
- 版本号是否递增
- 日期是否合理
- 修改内容是否清晰

**输出**：
- 变更记录完整性检查结果
- 缺失或异常的变更记录提示

## 最佳实践

### 生成PRD时

1. **提供清晰的需求描述**：包含核心功能、目标用户、使用场景

### 分析PRD时

1. **定期检查完整性**：确保PRD文档符合规范要求
2. **验证公共需求**：确保公共需求/能力分析完整
3. **检查竞品分析**：确保竞品分析有深度和广度
4. **关注KEP/KEI**：确保关键体验路径和指标明确

## 常见问题

**Q: 生成的PRD可以直接使用吗？**
A: 生成的PRD是基础框架，需要根据具体业务进行调整和完善。

**Q: 如何自定义PRD模板？**
A: 可以修改 [assets/prd_template.md](assets/prd_template.md) 来自定义模板。

**Q: 分析报告可以导出为Excel吗？**
A: 可以使用 `--format xlsx` 选项导出为Excel格式。

**Q: 支持哪些产品品类？**
A: 支持手机直板机、折叠机、平板、PC、智慧屏、座舱、手表等多种产品品类。

**Q: 公共需求分析是必须的吗？**
A: 是的，整体分析是必选的，受影响的相关功能分析是可选的。
