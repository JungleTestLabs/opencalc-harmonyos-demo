# 需求规格检查项

每个检查项都按 `枚举 → 分类 → 汇总 → 发现项` 的顺序执行。

## SP-01 需求可判定性 → D1

| 子检查 | 规则 | 违规示例 |
|--------|------|----------|
| 定量验收 | 必须包含数字、阈值、百分比或时间条件 | “快速响应” → 违规 |
| 模糊词检测 | 检测主观形容词或副词 | “友好的”“快速”“高性能”“稳定”“合理”“尽量”“适当” |
| 复合需求 | 一条需求同时包含多个独立能力 | “支持登录 AND 支持 SSO AND 记录审计日志” |
| 隐含前提 | 需求依赖未声明的前提 | “支持离线模式”，但未说明数据范围或同步策略 |

**输出指标：** 总数 N，可判定 M，占比 M/N；模糊词 K 个；复合需求 J 条。每条不可判定需求都要给出改写建议。

---

## SP-02 需求 9 类完整性 → D2

检查是否覆盖：功能、性能、可用性、安全、兼容性、约束、范围、用户角色、数据。

**输出指标：** 已覆盖类别数 / 9。每个缺失的关键类别都要给出补充建议。

---

## SP-03 需求 ID 与引用 → D5

每条需求必须有唯一 ID。不得重复、跳号或引用不存在的目标。优先级分布应合理；如果所有需求都是 “high”，说明没有真正排序。

---

## SP-04 术语一致性 → D3

同一概念必须使用同一术语。术语应在首次使用时定义，缩写应在首次出现时展开。

---

## SP-05 准确性 → D4

外部标准、协议和法规必须真实存在。性能基线应合理。技术约束不得自相矛盾。

---

## SP-06 可行性 → D6

NFR 之间不得冲突。NFR 的定量目标必须在物理和工程上可实现。资源约束应匹配功能范围，时间约束也应匹配功能范围。

---

## SP-07 可验证性 → D7

每条需求都应有清晰的二元验收条件。NFR 必须可测量，例如“高可用”不可测，“年可用率 ≥ 99.9%”可测。验收测量方法应明确。

---

## SP-08 IPD 增量需求规格兼容 → D2/D5/D7

仅当输入是 `delta-spec.md` 或明显的 IPD 增量需求规格时执行。

必须检查：
- 变更概述是否说明变更类型、关联特性和受影响的 DFR 约束
- 需求条目是否使用可追溯 ID，例如 `REQ-{特性前缀}-001` 或 `REQ-DFR-{特性前缀}-001`
- 场景分析是否覆盖目标、场景要素因子、关键场景因子和场景实例化
- 用例分析是否能从需求追溯到场景，再追溯到用例和验收路径
- 影响分析是否列出涉及模块、依赖关系和风险评估

如果 `delta-spec.md` 只有普通需求列表，缺少场景 / 用例 / 影响链路，应按 D2 和 D5 产生发现项。若需求无法转成验收路径，应按 D7 产生发现项。

---

## Skill 优化映射

当用户要求优化生成需求规格的 skill 时，按以下方式反推：

| 检查项 | 常见文档缺陷 | 对应 skill 缺陷 | skill 优化建议 |
|--------|--------------|----------------|----------------|
| SP-01 | 需求不可判定、模糊词多 | S3 生成约束问题 | 增加可判定表达规则、模糊词禁用清单、验收条件自检 |
| SP-02 | 需求类别缺失 | S2 输出结构问题 / S7 模板缺陷问题 | 固化 9 类需求覆盖清单，允许“不适用”但必须说明原因 |
| SP-03 | ID 重复、跳号、引用断裂 | S4 追溯链路问题 | 增加 ID 生成规则和引用完整性自检 |
| SP-04 | 术语漂移 | S6 自检缺失问题 | 增加术语表和术语一致性检查 |
| SP-05 | 编造标准、协议或约束 | S5 事实校验问题 | 增加外部事实需核实或标注待确认的规则 |
| SP-06 | NFR 与资源约束冲突 | S1 输入理解问题 / S5 事实校验问题 | 增加资源-目标一致性检查 |
| SP-07 | 验收条件不可测 | S3 生成约束问题 | 强制每条需求有二元验收条件 |
| SP-08 | 增量规格缺少场景、用例、影响链路 | S2 输出结构问题 / S4 追溯链路问题 | 在生成 `delta-spec.md` 时强制输出场景分析、用例分析和影响分析 |
