# 评分概览

仅在用户明确要求评分时使用。大多数审计场景下，优先给出维度级定性标签（强 / 可接受 / 弱 / 缺失），不要默认输出完整加权分。

## 评分维度

| 维度 | 核心问题 | 5 分标准 |
|------|----------|----------|
| D1 清晰性 | 内容是否无歧义、可判定？ | 可判定率 ≥ 95%，模糊词占比 < 2% |
| D2 完整性 | 主要必需内容是否存在？ | 覆盖率 ≥ 95% |
| D3 一致性 | 术语和陈述是否无矛盾？ | 矛盾数 = 0，术语变体 = 0 |
| D4 准确性 | 技术事实是否真实且适用？ | 无幻觉，引用的技术和协议均正确 |
| D5 追溯性 | 条目能否连接上游和下游文档？ | 链路完整、双向可追溯、无孤立项 |
| D6 可行性 | 系统能否在约束下实现？ | 所有 NFR 自洽，无不可实现需求 |
| D7 可验证性 | 是否能写出验收测试？ | 每个条目都有清晰验收条件 |
| D8 可维护性 | 系统后续演进是否代价可控？ | 模块化完整，版本策略明确 |

### 单文档关注点

- **spec.md**：D1 关注需求条目；D2 检查 9 类需求覆盖；D5 检查 ID 唯一性。
- **arch.md**：D1 关注组件职责；D2 检查技术栈理由；D6 检查 NFR 承接能力。
- **design.md**：D1 关注接口和算法是否能直接编码；D2 检查接口 7 要素和数据模型 4 要素覆盖。
- **proposal.md**：D2 关注提案章节覆盖；D5 检查前置分析、需求、影响范围和跳过策略追溯；D6 检查复杂度评估。
- **需求解析**：D2 关注结构化字段覆盖；D4 检查分类和领域映射准确性；D6 检查复杂度判断是否合理。
- **需求澄清**：D1 关注问题是否具体；D7 检查澄清结果是否可转成可执行需求；D5 检查问题是否追溯到模糊点。
- **代码仓理解**：D4 关注代码事实准确性；D5 检查结论是否有代码证据；D2 检查分析范围是否覆盖。

