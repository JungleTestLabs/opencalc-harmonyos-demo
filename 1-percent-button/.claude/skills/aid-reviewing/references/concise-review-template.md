# 轻量审阅模板

适用于默认模式下所有受支持文档类型的输出。

文档类型通过 `{DOC_KIND}` 参数区分，常见取值：`spec` | `arch` | `design` | `proposal` | `需求解析` | `需求澄清` | `代码仓理解`。

````md
# {DOC_KIND} 简要审阅

**文档:** {FILE_PATH}
**文档类型判断:** {DOC_TYPE_JUDGMENT}（置信度：{DOC_TYPE_CONFIDENCE}）
**作者意图匹配度:** {INTENT_FIT} / 5
**结论:** {INTENT_VERDICT}

## 1. 作者可能想表达的重点
{AUTHOR_INTENT_SUMMARY}

## 2. 关键问题（按 D1-D8 维度标注，只保留最重要的 3-5 条）

每个发现必须使用以下格式，确保可被自动化提取：

- **D{维度编号}**，{严重程度标记}：{具体位置，如 REQ-001 / NFR-003 / 某章节名}
  {一句话问题描述}

可用维度：D1 清晰性 | D2 完整性 | D3 一致性 | D4 准确性 | D5 追溯性 | D6 可行性 | D7 可验证性 | D8 可维护性
严重程度：🔴 严重（必须修复）| 🟡 重要（应该修复）| 🟢 轻微（建议修复）

示例：
- **D1**，🔴：NFR-001
  "系统应快速处理"使用模糊词"快速"，缺少定量响应时间指标
- **D7**，🟡：REQ-006
  "妥善处理库存不足情况"无明确验收标准，应指定具体错误码和用户反馈
- **D2**，🔴：安全性需求
  缺少安全需求类别，9 类需求覆盖率不足

## 3. 更合适的表达方向
{BETTER_FRAMING}

## 4. 最少确认问题
- {CONFIRM_Q1}
- {CONFIRM_Q2}

---

# 优化后的 {DOC_KIND} 文档

{OPTIMIZED_DOC_MD}

优化稿修复要求：
- 将所有检测到的 D1 模糊词替换为定量标准（如"快速" → "P99 ≤ 500ms"）
- 将 D7 不可验证的验收标准替换为二元判定条件
- 补充 D2 缺失的关键类别（以 Assumption 或 Reviewer suggestion 标注）
- 修复 D5 断裂的 ID 和引用链
- 删除 D4 不存在的引用
- 不得捏造原文没有的业务事实
````
