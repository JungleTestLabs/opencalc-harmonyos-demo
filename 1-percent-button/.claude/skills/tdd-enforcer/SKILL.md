# TDD Enforcer

## 概述

对每个开发任务强制执行 RED-GREEN-REFACTOR 循环。

**核心原则：没有失败的测试，就不写生产代码。**

**适用范围：**
- 新增功能
- 功能变更
- 缺陷修复（bugfix）
- 优化或重构

**例外（需用户明确同意）：**
- 纯 UI 样式调整（无逻辑）
- 配置文件变更
- 生成代码（自动生成的代码）

## The Iron Law

```
没有失败的测试 → 不写生产代码
```

先写了代码？删掉，从测试开始重来。

**没有例外：**
- 不要保留"作为参考"
- 不要"边写测试边改代码"
- 不要"回头补测试"
- 删除就是删除

## RED-GREEN-REFACTOR 流程

每个 task 必须按以下循环执行：

```
RED（写失败测试）→ 验证失败 → GREEN（最小代码通过）→ 验证通过 → REFACTOR（重构）→ 验证仍通过
```

### RED - 写失败测试

针对当前 task 的验收标准，写一个最小测试：
- 一个测试只验证一个行为
- 测试名称清晰描述期望行为
- 使用真实代码，避免不必要的 mock

```typescript
// 示例：bugfix - 空邮箱被接受
test('rejects empty email', async () => {
  const result = await submitForm({ email: '' });
  expect(result.error).toBe('Email required');
});
```

### 验证 RED

运行测试，确认：
- 测试**失败**（不是报错）
- 失败原因是功能缺失（不是拼写错误）
- 失败信息符合预期

```bash
hvigorw test@entry
```

**测试通过了？** 说明你在测试已有行为，修改测试。
**测试报错了？** 修复语法错误，重新运行直到正确失败。

### GREEN - 最小代码通过

写最简单的代码让测试通过：
- 不添加额外功能
- 不重构其他代码
- 不"顺便改进"

### 验证 GREEN

运行测试，确认：
- 当前测试通过
- 其他测试仍然通过
- 无错误、无警告

```bash
hvigorw test@entry
```

**测试失败？** 修改代码，不修改测试。

### REFACTOR - 重构

测试全绿后：
- 消除重复
- 改善命名
- 提取辅助函数

保持测试绿色，不添加新行为。

### 循环

当前 task 的所有验收标准都有对应测试且通过后，进入下一个 task。

## Bugfix 专用流程

缺陷修复必须遵循 TDD：

1. **RED**：写一个能复现 bug 的测试（测试应失败）
2. **验证 RED**：确认测试失败且失败原因就是该 bug
3. **GREEN**：修复 bug，使测试通过
4. **验证 GREEN**：确认测试通过且无回归
5. **REFACTOR**：清理修复代码

**绝不允许**：修复 bug 后不写测试。测试证明修复有效，并防止回归。

## 为什么必须先写测试

**"先写代码再补测试不是一样吗？"**

不一样。后补的测试立即通过，你永远不知道它是否真的在测你想测的东西。先写测试、看到失败，证明测试确实在检测缺失的功能。

**"手动测试过了"**

手动测试是临时的。无法重复、无法回归验证、无法证明覆盖了所有边界。

**"删掉已写的代码太浪费"**

沉没成本谬误。保留未经测试验证的代码才是真正的浪费——它是技术债。

## 常见借口与应对

| 借口 | 现实 |
|------|------|
| "太简单了不需要测试" | 简单代码也会出错，写测试只需 30 秒 |
| "我先写代码再补测试" | 后补的测试立即通过，证明不了任何事 |
| "后补测试目的一样" | 后补测试回答"代码做了什么"，先写测试回答"代码应该做什么" |
| "手动测试过了" | 手动测试无法重复、无法回归验证 |
| "TDD 太慢了" | TDD 比事后调试更快 |
| "这次特殊，跳过一次" | 每次都"特殊"，纪律没有例外 |
| "删掉已写的代码太浪费" | 沉没成本谬误，保留未验证代码才是浪费 |
| "保留作为参考，测试先写" | 你会"适配"它，那就是后补测试。删除就是删除 |
| "需要先探索一下" | 可以。探索完后丢弃探索代码，从 TDD 开始 |

## Red Flags - 立即停止并重来

出现以下情况时，必须停止当前操作，删除已写代码，从 RED 开始：

- 先写了代码再写测试
- 测试立即通过（没看到失败）
- 无法解释测试为什么失败
- 说"回头补测试"
- 说"这次跳过"
- 说"保留作为参考"
- 说"后补测试目的一样"
- 说"TDD 太教条了，我务实一点"
- 说"这次不一样因为..."

**以上所有情况 = 删除代码，从 TDD 重新开始。**

## 卡住时怎么办

| 问题 | 解决方案 |
|------|---------|
| 不知道怎么写测试 | 先写断言（expect），再倒推测试结构 |
| 测试太复杂 | 说明设计太复杂，简化接口 |
| 必须 mock 所有东西 | 代码耦合太紧，使用依赖注入 |
| 测试 setup 太大 | 提取辅助函数；仍然复杂则简化设计 |

## 完成检查清单

每个 task 完成前，确认：

- [ ] 每个验收标准都有对应测试
- [ ] 每个测试都先看到失败（RED）
- [ ] 每个测试失败原因是功能缺失（不是拼写错误）
- [ ] 写了最小代码使测试通过（GREEN）
- [ ] 所有测试通过
- [ ] 重构后测试仍通过（REFACTOR）
- [ ] 无错误、无警告

无法全部勾选？说明跳过了 TDD，从头开始。
