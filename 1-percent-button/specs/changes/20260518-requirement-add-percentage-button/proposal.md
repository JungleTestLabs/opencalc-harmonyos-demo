# 需求提案 (Proposal) — 百分号按钮

> 变更：20260518-requirement-add-percentage-button
> 类型：New Requirement（新增功能）
> 创建日期：2026-05-18

## 1. 原始需求

> 加一个百分号按钮，输入50+10%自动算出55

## 2. 需求解析报告（rq-parse 产物）

### 2.1 意图分类

- **类型**：`新增功能`
- **一句话概括**：在计算器按钮区新增 `%` 按钮，使表达式中可输入百分号，并支持「上下文相关」语义（`a+b%` 等价于 `a + a*(b/100)`）。

### 2.2 领域映射

| 领域 | 涉及内容 |
|------|---------|
| **UI/交互** | 在 `ButtonGrid` 中新增一个 `%` 按钮；用户按下时把字符 `%` 追加到当前表达式 |
| **数据/计算** | （无需变更）`Expression.getPercentString()` 已实现智能百分号语义；引擎层无改动 |

### 2.3 影响层次

| 层次 | 文件 | 变更类型 | 说明 |
|------|------|---------|------|
| View | `entry/src/main/ets/pages/CalculatorPage.ets` | `modify` | 在 `@Builder ButtonGrid()` 中新增 `%` 按钮；可能调整一行 4 列 → 5 列，或将 `%` 替换/插入到某固定位置 |
| ViewModel | （同上：CalculatorPage 状态）| `extend` | 复用现有 `onOp(op: string)` 事件，传入 `'%'` 即可 |
| Model | — | — | 不涉及 |
| 数据层 | — | — | 不涉及 |
| 基础设施 | — | — | 不涉及 |

**关键代码现状**（已在仓库中验证）：

- `entry/src/main/ets/calculator/Expression.ets:14-17` 已在 `getCleanExpression()` 中对 `%` 进行预处理
- `entry/src/main/ets/calculator/Expression.ets:169-255` `getPercentString()` 已实现以下智能语义：
  - `a + b%` → `a + a*(b/100)`（验证：`50 + 10%` = `50 + 50*(10/100)` = `55` ✓）
  - `a - b%` → `a - a*(b/100)`
  - `a * b%` → `a * (b/100)`
  - `a / b%` → `a / (b/100)`
  - 支持括号子表达式、阶乘前置等组合
- `Expression.ets:58-62` `addMultiply()` 已处理 `%` 后接数字自动插 `*`

### 2.4 关键实体

| 类别 | 实体 |

| **页面实体** | `CalculatorPage.ets` 的 `ButtonGrid` `@Builder`（行 272-285） |

| **事件实体** | 用户点击 `%` 按钮 → `expression` 字符串追加 `%` 字符 |
| **数据实体** | 无新增 |

### 2.5 模糊点列表

| # | 类别 | 描述 | 严重程度 | 备选答案 |
|---|------|------|---------|---------|
| 1 | 交互模糊 | **`%` 按钮的具体位置**：现有 5 行 × 4 列布局已被数字 / `AC ( ) ÷` / `× - + =` 占满，需决定如何放入新按钮 | important | A) 替换 `(` 或 `)`（不推荐，丢失功能）；B) 在 `AC ( ) ÷` 行用 `%` 替换 `AC`（重排）；C) 新增一行；D) **推荐：将首行改为 `AC % ⌫ ÷`，把现有 `(` `)` 挪到科学模式或合并；E) 推荐折衷：把首行改为 5 列 `AC ( ) % ÷` ，仅该行 5 列** |
| 2 | 范围模糊 | **按钮在哪些模式可见**：基础模式 / 科学模式 / 两者都可见 | important | 推荐：两种模式下都可见（百分号是基础运算需求） |
| 3 | 行为模糊 | **回填功能是否影响**：例如历史记录回填、退格等是否需要适配 | nice-to-have | 无需额外适配，`%` 当作普通字符处理即可 |
| 4 | 视觉模糊 | **按钮配色风格**：跟随运算符颜色 `getOp()` / 数字按钮颜色 `getBtnBg()` / 功能按钮颜色 `getFuncBg()` | nice-to-have | 推荐：跟随运算符颜色 `getOp()`，因为 `%` 在功能上属于「运算符家族」 |

> **存在 important 级别模糊点**，将在 step3 通过 rq-clarify 与用户确认。

### 2.5.1 澄清结果（step3 rq-clarify 产物，2026-05-18）

| 模糊点 | 用户决策 |
|--------|---------|
| #1 按钮位置 | **首行 5 列：`AC ( ) % ÷`**（其他行保持 4 列布局） |
| #2 可见模式 | **基础模式 + 科学模式都可见** |
| #3 回填/退格 | 沿用默认（按普通字符处理） |
| #4 视觉风格 | **运算符风格**（蓝色 `getOp()`，与 `+` `-` `×` `÷` 一致） |

**布局示意**（澄清后）：

```
AC  (   )   %   ÷       ← 首行 5 列（每按钮宽 ≈ 18%）
7   8   9       ×       ← 4 列（每按钮宽 ≈ 22%）
4   5   6       -
1   2   3       +
0   .   ⌫       =
```

> 说明：因首行从 4 列变为 5 列，按钮宽度从 22% 调整为约 18%，仅影响首行视觉。其他行保持不变。

### 2.6 复杂度评估

- **等级**：**S（简单）**
- **理由**：
  - 改动模块数：1 个（`CalculatorPage.ets` 仅）
  - 代码增量：预计 < 30 行（一个 `@Builder BtnPct` + 一行布局调整）
  - 无新数据模型、无架构变更、无新接口
  - 核心计算逻辑（`Expression.getPercentString`）已完整实现，无需触碰

## 3. 验收准则（初版，可在 step3 微调）

1. 在计算器主界面（基础模式 + 科学模式）能看到一个 `%` 按钮
2. 点击 `%` 后，当前表达式字符串末尾追加字符 `%`
3. 输入 `50+10%` 后按 `=`，结果显示 `55`
4. 输入 `50-10%` 后按 `=`，结果显示 `45`
5. 输入 `50*10%` 后按 `=`，结果显示 `5`
6. 输入 `50/10%` 后按 `=`，结果显示 `500`
7. 输入 `10%` 后按 `=`，结果显示 `0.1`
8. 现有 `AC` / 退格 / 历史记录 / `=` 功能不受影响（回归）

- step3（rq-clarify）将聚焦于按钮位置与可见性，避免对实现细节过早决策
