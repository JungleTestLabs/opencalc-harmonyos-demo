# PRD 完整性检查规则

本文档定义基于终端V2.1模板的PRD文档完整性验证规则。

## 章节完整性矩阵

| 章节 | 必需 | 必需子章节 | 验证标准 |
|------|------|-----------|----------|
| 文档信息 | ✅ 是 | 5（文档名称、版本号、创建日期、最后更新、文档状态） | 所有子章节存在且有内容 |
| 文档变更记录 | ✅ 是 | ≥1条记录 | 至少包含初始版本记录 |
| 产品需求 | ✅ 是 | 5（需求来源、需求背景、需求价值分析、需求适用范围、需求描述） | 所有子章节存在 |
| 需求适用范围分析 | ✅ 是 | 产品品类+发布范围表格 | 至少填写一项产品品类 |
| 需求描述 | ✅ 是 | 4（参与者、前置条件、业务流程、功能描述） | 所有子章节存在 |
| KEP定义 | ✅ 是 | ≥1个KEP | 每个KEP有阶段、场景、Actor、场景名称、用例等 |
| KEI定义 | ✅ 是 | ≥1个KEI | 每个KEI有评估体系、维度、评估类型、评估标准 |
| 公共需求/能力交互影响分析 | ✅ 是 | 整体分析表格 | 至少分析国际化、安全隐私、全局业务 |
| 非功能需求 | ✅ 是 | 3（性能要求、安全要求、兼容性要求） | 所有子章节存在 |
| 附录 | ✅ 是 | 3（术语表、参考资料、相关文档） | 所有子章节存在 |
| 审批记录 | ✅ 是 | 4个角色 | 产品经理、技术负责人、测试负责人、项目经理 |

## 详细验证规则

### 1. 文档信息

#### 1.1 基本信息

**必需内容**：
- 文档名称
- 版本号（格式：VX.X）
- 创建日期（YYYY-MM-DD）
- 最后更新日期
- 文档状态（草稿/评审中/已批准）

**验证**：
- [ ] 包含所有5个基本信息项
- [ ] 版本号格式正确
- [ ] 日期格式正确
- [ ] 文档状态已选择

### 2. 文档变更记录

**必需内容**：
- 版本号
- 日期
- 修改人
- 修改内容
- 备注

**验证**：
- [ ] 至少包含一条记录（初始版本）
- [ ] 每条记录包含5个字段
- [ ] 版本号与文档信息一致

### 3. 产品需求

#### 3.1 需求来源

**必需内容**：
- 需求提出部门
- 接口人
- 需求分类（运营需求/优化需求/用户反馈需求/规划需求）

**验证**：
- [ ] 需求来源明确
- [ ] 接口人信息完整
- [ ] 需求分类已选择

#### 3.2 需求背景及业务定位

**必需内容**：
- 需求的原始诉求
- 需要解决的问题
- 期望达到的目标

**验证**：
- [ ] 背景描述清晰
- [ ] 问题定位准确
- [ ] 目标明确

#### 3.3 需求价值分析

**必需内容**：
- KPI提升
- 用户体验改善
- 用户满意度提升
- 减少运营工作量等

**验证**：
- [ ] 至少列出1个价值点
- [ ] 价值描述具体可衡量

#### 3.4 需求适用范围分析

**必需内容**：
- 产品品类（手机、平板、PC、智慧屏、座舱、手表等）
- 发布范围（中国区、亚非拉、欧洲等）

**验证**：
- [ ] 至少选择1个产品品类
- [ ] 至少选择1个发布范围
- [ ] 涉及的项已填写"涉及"
- [ ] 不涉及的项已填写"不涉及"

#### 3.5 目标用户分析（可选）

**必需内容**（如果填写）：
- 用户基本信息
- 产品/服务使用情况
- 动机与关注点
- 需求及痛点
- 用户原声

**验证**：
- [ ] 用户画像清晰
- [ ] 痛点描述具体

#### 3.6 竞品分析（可选）

**必需内容**（如果填写）：
- 至少2个竞品
- 典型场景/规格对比表格
- 优劣势总结
- 竞争力构建计划

**验证**：
- [ ] 竞品数量≥2个
- [ ] 对比维度清晰
- [ ] 分析结论明确

### 4. 需求描述

#### 4.1 需求参与者（角色）

**必需内容**：
- 最终用户

**验证**：
- [ ] 参与者角色已列出

#### 4.2 前置条件

**必需内容**：
- 设备要求
- 系统版本要求
- 验收范围

**验证**：
- [ ] 设备要求已明确
- [ ] 系统版本已指定
- [ ] 验收范围清晰

#### 4.3 业务流程

**必需内容**：
- 正常业务流程
- 异常业务流程
- 流程图

**验证**：
- [ ] 正常流程完整
- [ ] 异常流程已考虑
- [ ] 流程图清晰

#### 4.4 业务功能描述

**必需内容**：
- 功能描述表格
- 每个功能有IR需求编号
- 每个功能有描述
- 界面示意图（UI低保真）

**验证**：
- [ ] 功能表格完整
- [ ] IR编号规范
- [ ] 描述清晰
- [ ] 界面示意图已提供

### 5. 关键体验路径（KEP）

**必需内容**：
- 阶段
- 场景类型
- 涉及Actor
- KEP描述
- 场景名称
- 场景编号
- 用例名称
- 用例编号
- IR需求编号

**验证**：
- [ ] 至少定义1个KEP
- [ ] KEP字段完整
- [ ] 编号规范
- [ ] 与IR需求关联

### 6. 关键体验指标（KEI）

**必需内容**：
- 评估体系
- 维度
- 评估类型
- 是否必选
- 评估标准
- IR需求编号
- 功能名称
- 功能编码

**验证**：
- [ ] 至少定义1个KEI
- [ ] 评估标准可测量
- [ ] 与KEP关联

### 7. 公共需求/能力交互影响分析

#### 7.1 整体分析（必选）

**必需内容**：
- 国际化（中国大陆、全球）
- 安全隐私（儿童账号、已收集个人信息清单、隐私空间、停止服务）
- 全局业务（无障碍、适老化、退账号、注销账号、OOBE、克隆备份、旋转屏、分屏、外屏适配、自由多窗、双升单、三方应用关联启动管理、单回退双、小艺问答操控、统一交互）

**验证**：
- [ ] 所有能力项已评估
- [ ] 涉及/不涉及已选择
- [ ] 涉及的理由已填写
- [ ] 遵循的规范已列出

#### 7.2 受影响的相关功能分析（可选）

**必需内容**（如果启用）：
- 公共需求/能力名称
- 功能点
- 是否支持
- 说明

**验证**：
- [ ] 涉及的功能已列出
- [ ] 支持情况已明确
- [ ] 说明清晰

### 8. 运营策略

**必需内容**：
- 产品/特性/需求名称
- 用户触点（整体、端内、门店体验及演示等）
- 运营目标（北极星指标）
- 运营策略

**验证**：
- [ ] 至少定义1个触点
- [ ] 北极星指标明确
- [ ] 运营策略具体

### 9. 非功能需求

#### 9.1 性能要求

**必需内容**：
- 响应时间
- 并发用户数
- 数据处理量

**验证**：
- [ ] 至少指定1个性能指标
- [ ] 指标有具体数值
- [ ] 指标可测量

#### 9.2 安全要求

**必需内容**：
- 数据加密
- 权限控制
- 其他安全需求

**验证**：
- [ ] 至少列出1项安全要求
- [ ] 要求具体可实施

#### 9.3 兼容性要求

**必需内容**：
- 平台
- 版本要求
- 备注

**验证**：
- [ ] 至少列出1个平台
- [ ] 版本要求明确

### 10. 附录

#### 10.1 术语表

**必需内容**：
- 术语
- 定义

**验证**：
- [ ] 至少定义1个术语
- [ ] 定义清晰

#### 10.2 参考资料

**必需内容**：
- 相关文档列表

**验证**：
- [ ] 参考资料已列出

#### 10.3 相关文档

**必需内容**：
- 相关文档名称

**验证**：
- [ ] 相关文档已列出

### 11. 审批记录

**必需内容**：
- 产品经理
- 技术负责人
- 测试负责人
- 项目经理

**验证**：
- [ ] 所有4个角色已列出
- [ ] 姓名已填写
- [ ] 日期已填写
- [ ] 签名已填写

## 完整性评分

### 分数计算

为每个章节分配分数：

- **2分**：完整包含所有必需内容
- **1分**：部分完整（缺少部分必需内容）
- **0分**：缺失或严重不完整

### 总分

```
总分 = 所有章节分数之和
最高分 = 20分（10个章节 × 2分）
```

### 完整性等级

| 分数范围 | 等级 | 说明 |
|----------|------|------|
| 18-20 | 优秀 | 所有必需内容都存在 |
| 15-17 | 良好 | 少量缺口，可接受 |
| 12-14 | 一般 | 明显缺口，需要完善 |
| 0-11 | 差 | 主要缺口，需要修订 |
