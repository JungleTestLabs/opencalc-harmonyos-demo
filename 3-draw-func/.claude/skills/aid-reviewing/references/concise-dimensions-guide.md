# 轻量 D1-D8 缺陷检测指南

在 `default concise` 模式下使用，提供 8 个缺陷维度的快速检测清单。每个维度给出：核心问题 + 文档类型对应的具体检查点。

## D1 清晰性

**核心问题**：需求/设计是否可判定、无模糊词？

| 文档类型 | 检查点 |
|----------|--------|
| spec | 每条需求包含定量标准（数字/阈值/百分比/时间）；无"快速""稳定""合理""适当""尽量"等主观词 |
| arch | 组件责任描述具体（避免"处理业务逻辑"等空泛表述） |
| design | 接口和算法描述可直接指导编码，无"适当处理""合理设计"等模糊表述 |
| 需求提案 | 需求概述、功能需求、影响范围和复杂度判断都有明确依据 |
| 需求解析 | 意图分类、领域映射、影响层次和复杂度判断有明确依据 |
| 需求澄清 | 问题指向具体模糊点，避免"再详细说说"这类泛问 |
| 代码仓理解 | 代码仓结论指向具体文件、模块、接口或调用链 |

**检测信号**：搜索主观形容词/副词 → 标记为 D1

## D2 完整性

**核心问题**：关键章节/类别是否缺失？

| 文档类型 | 必检类别 |
|----------|----------|
| spec | 9 类：功能、性能、可用性、安全、兼容性、约束、范围、角色、数据 |
| arch | 组件列表、技术栈（名称+版本+理由）、NFR 策略、设计决策 |
| design | 接口 7 要素、数据模型 4 要素、错误处理、状态机 |
| 需求提案 | 需求概述、需求解析、需求澄清、需求详述、影响范围、复杂度评估、待解决问题 |
| 需求解析 | 原始需求、意图分类、领域映射、影响层次、关键实体、模糊点、复杂度 |
| 需求澄清 | 模糊点、问题批次、用户决策、待定假设、功能边界 |
| 代码仓理解 | 需求类别、重点分析范围、架构、接口、影响范围、风险、测试策略 |

**检测信号**：逐类对照 → 缺失类别标记为 D2

## D3 一致性

**核心问题**：同一概念是否使用同一术语？

| 文档类型 | 检查点 |
|----------|--------|
| spec | 术语表内定义 vs 正文使用一致；缩写首次使用有展开 |
| arch | 与需求规格术语一致；新增技术术语有定义 |
| design | 与需求规格和架构设计术语一致；组件名、接口名跨文档匹配 |
| 需求提案 | 需求概述、需求解析、需求澄清和需求详述中的术语一致 |
| 需求解析 | 需求类别、领域名称、实体名称前后一致 |
| 需求澄清 | 模糊点、问题、用户决策、最终边界之间不矛盾 |
| 代码仓理解 | 模块名、文件名、接口名与代码仓实际命名一致 |

**检测信号**：同一实体的不同称呼 → 标记为 D3

## D4 准确性

**核心问题**：引用的标准/组件/协议是否真实存在且适用？

| 文档类型 | 检查点 |
|----------|--------|
| spec | 外部标准（PCI-DSS、ISO 等）存在且级别正确 |
| arch | 第三方组件/框架存在，版本正确，协议匹配场景 |
| design | 引用的库/API/协议存在且适用 |
| 需求提案 | 不把未确认的假设、平台限制或代码事实写成已确认事实 |
| 需求解析 | 平台限制、领域映射、复杂度判断不编造事实 |
| 需求澄清 | 不把未确认选项写成用户已确认事实 |
| 代码仓理解 | 所有代码仓结论都有真实文件、符号或调用关系支撑 |

**检测信号**：不存在的标准编号（如 ISO-99999）、不存在的组件名 → 标记为 D4

## D5 追溯性

**核心问题**：需求 ID、引用链是否完整无断裂？

| 文档类型 | 检查点 |
|----------|--------|
| spec | 每条需求有唯一 ID；无重复或跳号；相互引用指向存在的目标 |
| arch | 每个组件可追溯到至少一个需求规格中的需求；每个需求规格中的需求有对应组件 |
| design | 每个设计接口属于某个架构组件 |
| 需求提案 | 功能需求、影响范围、复杂度和跳过策略能追溯到前置分析 |
| 需求解析 | 模糊点能追溯到原始需求中的具体表达 |
| 需求澄清 | 每个澄清问题能追溯到一个 blocker 或 important 模糊点 |
| 代码仓理解 | 每个影响范围结论能追溯到具体代码证据 |

**检测信号**：ID 缺失/重复/断裂、孤立的引用 → 标记为 D5

## D6 可行性

**核心问题**：在给定约束下目标是否可实现？

| 文档类型 | 检查点 |
|----------|--------|
| spec | NFR 目标与资源约束不冲突（如"单实例 2 核 4G"vs"支撑 10K 并发"） |
| arch | 技术选型支持 NFR 目标；无单点瓶颈；扩展方案可行 |
| design | 算法复杂度在性能预算内；数据模型无查询瓶颈 |
| 需求提案 | 复杂度等级、影响模块数、架构变更和数据模型变更判断自洽 |
| 需求解析 | 复杂度等级与领域数量、影响层次、风险数量匹配 |
| 需求澄清 | 澄清问题数量可控，能在有限轮次内收敛 |
| 代码仓理解 | 分析深度与需求类别匹配，不做无关全量扫描 |

**检测信号**：资源不足 + 高目标并存 → 标记为 D6

## D7 可验证性

**核心问题**：每条需求/设计是否有明确的验收标准？

| 文档类型 | 检查点 |
|----------|--------|
| spec | 每条需求有二元验收标准；NFR 指标可测量 |
| arch | 有测试策略（单元/集成/E2E）；可观测性（日志/指标/追踪） |
| design | 接口暴露测试控制点；状态机所有状态可构造 |
| 需求提案 | 功能需求有验收条件，跳过策略能指导后续制品生成 |
| 需求解析 | 输出能明确判断是否可进入澄清或生成需求规格 |
| 需求澄清 | 澄清结果能转成明确需求边界和验收条件 |
| 代码仓理解 | 测试验证策略覆盖受影响模块和关键风险 |

**检测信号**："妥善处理""适当处理""及时"等无测量方法的表述 → 标记为 D7

## D8 可维护性

**核心问题**：系统是否易于演进？

| 文档类型 | 检查点 |
|----------|--------|
| spec | 不涉及（跳过） |
| arch | 组件可独立部署；版本兼容策略；扩展点明确 |
| design | 接口向后兼容策略；配置外置；错误码预留扩展空间；无硬编码 |
| 需求提案 | 待解决问题、假设和跳过策略清楚，后续变更可追踪 |
| 需求解析 | 分类、领域、实体表达便于后续复用和拆分 |
| 需求澄清 | 决策和假设记录清楚，后续变更可追踪 |
| 代码仓理解 | 分析结果结构化，可被后续 spec / arch / design 复用 |

**检测信号**：硬编码值、紧耦合、无版本策略 → 标记为 D8

---

## 快速检测流程

轻量模式下按以下顺序扫描：

1. **D1 扫描**：全文搜索模糊词
2. **D2 扫描**：对照必检类别表
3. **D5 扫描**：检查 ID 唯一性和引用链
4. **D7 扫描**：检查验收标准是否可测量
5. **D4 扫描**：核实外部引用
6. **D3 扫描**：检查术语一致性
7. **D6 扫描**：检查资源-目标匹配
8. **D8 扫描**：检查是否便于后续演进、复用和变更追踪；需求规格可跳过

## 发现项输出格式

每个发现必须按以下格式输出，确保可被解析器提取：

```
- **D{编号}**，{严重程度}：{位置描述}
  {具体问题描述}
```

严重程度三选一：
- 🔴 严重（必须修复）
- 🟡 重要（应该修复）
- 🟢 轻微（建议修复）
