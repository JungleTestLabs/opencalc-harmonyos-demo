⏺ 根因有三层，缺一不可：

  1. DevEco 6.0.2 的 SDK 是嵌套布局
  新装出来的 SDK 把 ets/js/native/previewer/toolchains 这 5 个组件全塞进
  sdk/default/openharmony/ 子目录，顶层只有一个
  sdk-pkg.json（path="HarmonyOS-6.0.2"）。hvigor 默认扁平扫描只能扫到这 1
  个根条目，看不到 5 个真正的组件。

  2. 必须走 _parsingPlatforms 分支
  _mapComponentsByPath 里有个开关：当 HOS_CONFIG.pathVersionMapper["6.0.2"]
  非空字符串 时，才会走 _parsingPlatforms(o, e)，那个函数会用 PlatformSdks._base
   = ["ets","js","native","previewer","toolchains"] 列表 +
  getSdkType()="openharmony" 拼出 sdk/default/openharmony/<comp> 这 5
  个路径，把它们加进 map。

  重装 DevEco 后，磁盘上的 hos-config.json 实际上已经把 
  pathVersionMapper["6.0.2"] 设成了 "HarmonyOS 
  NEXT2"（非空），这层条件本来就满足。

  3. 但是 hvigor daemon 缓存了启动时的 HOS_CONFIG
  sync 命令带 --daemon，复用了一个 7:29PM 就启动的老 daemon —— 它的 Node require
   缓存里还是很久以前的 hos-config.json，那时候 pathVersionMapper["6.0.2"]
  还是空字符串，所以一直走错分支，map 里永远只有 1 个 entry。

  pkill 强杀 daemon → DevEco 拉起新 daemon → 新 daemon 重新 require 磁盘上的
  hos-config.json → 拿到非空值 → 走 _parsingPlatforms → map 里扫到 5 个组件 →
  checkComponentExistence(toolchains,ets,js,native,previewer) 通过。
